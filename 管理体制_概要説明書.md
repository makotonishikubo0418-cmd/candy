# candy Markdown管理体制 概要説明書

- 目的: candyでユーザーが希望した管理体制の全体像と狙いを簡潔に残す
- 文書の役割: 管理体制の狙いと設計思想の概要
- 詳細の正本: `README.md` に記載する各管理文書
- 状態: 確定（candy管理体制の概要）
- 更新日: 2026-07-16

## 1. 希望していること

candyのページ制作、生成ツール、GitHub反映、素材管理、複数Codex作業が乱雑にならないように、管理体制を先に整える。

目指す状態は次のとおり。

- 必要な確認を省かず、無駄な読込み・調査・報告だけをなくす
- 簡単な作業は簡潔に、重大な作業は慎重に扱う
- 複数のCodexが、同じファイルを同時に変更しない
- 誰が、何を、どこまで進めたか分かる
- 全体計画、現在地、問題点、課題、次の作業を見失わない
- 修正のたびに似たコードや上書きを増やさない
- 仕様、現在状態、過去の報告を混ぜない
- 報連相を行い、未確認のことを完成として扱わない
- 横展開しても耐えられる構成を維持する

ここでいう「最短」は、取り急ぎや手抜きではない。必要な工程は行い、不要な工程だけを省くという意味である。

## 2. 管理体制の基本設計

すべてを1つの巨大な `AGENTS.md` へ詰め込まず、入口を短くして、作業に必要な指示と資料だけを読む構成にする。

```text
AGENTS.md
  ↓
README.md
  ↓
今回必要な管理文書またはHP runbook
  ↓
対象のコード・仕様・素材・環境
```

この導線により、簡単な作業まで大量の資料を読む非効率を防ぎながら、必要な規則を読み落とさないようにする。

## 3. Markdownの責任分離

共通ルール、全体計画、仕様、コード構成、Codex間連絡、個別Task履歴を、それぞれ別の正本へ分ける。

実務上の正本一覧とファイルへの導線は `README.md` だけで管理し、この説明書には重複して持たない。

## 4. 作業の流れ

1. `AGENTS.md` で共通ルールを確認する。
2. `README.md` で今回読む正本を決める。
3. 稼働中の予約、関連する進行中Task、自分宛ての引継ぎ、対象の既存変更を確認する。
4. 重複または依存がある場合は、対象分割、順番変更、引継ぎを決める。
5. Taskと対象ファイルを予約する。
6. 既存の正本、依存関係、変更禁止範囲を確認する。
7. 既存の正本へ、小さく整合した変更を行う。
8. 差分、構文、品質ゲート、必要な実画面を確認する。
9. 実差分、確認済み、未確認、残件、引継ぎ先を報告する。
10. 予約を解除し、必要な場合だけCommit、Push、deployへ進む。

## 5. 防ぎたい問題と対策

| 防ぎたい問題 | 管理体制での対策 |
|---|---|
| AGENTSが長く、簡単な作業まで遅い | `AGENTS.md` は導線中心にする |
| 複数Codexが同じ場所を触る | Task IDとファイル予約で同時編集を防止する |
| 全体の現在地が分からなくなる | `PROJECT_STATUS.md` へ現在地・問題・次作業を集約する |
| 報告が散らばり、引継ぎできない | `CODEX_COMMUNICATION.md` と `TASK_LOG.md` で役割を分ける |
| 修正のたびにコードが汚くなる | 1責任1正本、既存経路への統合、末尾上書き禁止を徹底する |
| 仕様が文書ごとに違う | 内容ごとに正本を1つ決め、重複記載しない |
| 未確認なのに完成扱いされる | 確定・ユーザー申告・実装確認済み・未確認を区別する |
| 報連相が多すぎて作業を妨げる | 開始、重要な変更・障害、終了の必要な報告に絞る |

## 6. 完成イメージ

この管理体制は、文書を増やすこと自体が目的ではない。

必要な人またはCodexが、必要なときに、必要な正本へ最短で到達し、他の作業を壊さず、後から見ても判断と変更理由が分かる状態を作るためのものである。

維持するものは次の4点である。

1. 効率的で最短
2. 高品質で安全
3. 修正してもコードが汚くならない
4. 複数Codexでも報連相と責任範囲が崩れない

## 7. この説明書の扱い

この文書はcandy管理体制の概要説明書である。

実際の作業判断では、`AGENTS.md`、`README.md`、各管理文書の正本を使用する。詳細な運用規則を変更する場合は、この説明書だけを変更せず、該当する正本を更新する。
